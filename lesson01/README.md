# Блок 1. Инфраструктура и Docker

## Что такое Docker и какие задачи он решает?

Представьте ситуацию: вам нужно быстро развернуть базу данных для анализа логов, потом добавить систему визуализации, а через неделю всё это удалить, потому что расследование завершено. Раньше это означало бы часы на установку, настройку зависимостей и мучительную очистку системы. Docker решает эту проблему — он позволяет запускать приложения в изолированных контейнерах, которые легко создать и так же легко удалить.

Docker появился в 2013 году и быстро стал стандартом индустрии. Причина проста: он устранил проблему "у меня на машине работает". Контейнер содержит всё необходимое для работы приложения — код, библиотеки, настройки. Если контейнер работает у вас, он будет работать и у коллеги.

## Зачем Docker специалисту по информационной безопасности?

Docker стал стандартом для развёртывания программного обеспечения, в том числе инструментов ИБ. Практически все современные решения — TheHive, MISP, Wazuh, Security Onion, Elastic Stack — распространяются как Docker-образы.

Понимание контейнеризации необходимо для:

- __Развёртывания инфраструктуры мониторинга.__ SIEM-системы, сборщики логов, визуализаторы — всё это проще запустить в контейнерах, чем устанавливать вручную.
- __Изоляции инструментов анализа.__ При расследовании инцидента важно, чтобы инструменты не влияли друг на друга и на хостовую систему. Контейнер создаёт изолированную среду.
- __Воспроизводимости окружения.__ Когда коллега или следователь хочет повторить ваш анализ, достаточно передать Dockerfile — результат будет идентичным.
- __Быстрого развёртывания honeypot.__ Контейнеры позволяют за минуты поднять ловушку для злоумышленников и так же быстро её уничтожить.

Docker также активно используется в разработке. Dev Containers — это способ описать окружение разработчика в виде контейнера. Вы открываете проект, и VS Code или Cursor автоматически разворачивает контейнер со всеми нужными инструментами. Это особенно полезно, когда в команде работают люди с разными операционными системами.

## Основы Docker

### Контейнер vs виртуальная машина

Виртуальная машина эмулирует полный компьютер с собственной ОС. Контейнер использует ядро хостовой системы, изолируя только процессы и файловую систему. Это делает контейнеры легче и быстрее.

```
┌─────────────────────────────────────┐
│           Виртуализация             │
├─────────┬─────────┬─────────────────┤
│  App A  │  App B  │     App C       │
├─────────┼─────────┼─────────────────┤
│ Guest OS│ Guest OS│    Guest OS     │
├─────────┴─────────┴─────────────────┤
│            Hypervisor               │
├─────────────────────────────────────┤
│              Host OS                │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│          Контейнеризация            │
├─────────┬─────────┬─────────────────┤
│  App A  │  App B  │     App C       │
├─────────┴─────────┴─────────────────┤
│           Docker Engine             │
├─────────────────────────────────────┤
│              Host OS                │
└─────────────────────────────────────┘
```

### Образ и контейнер

- __Образ (image)__ — это шаблон, содержащий файловую систему и метаданные. Образ неизменяем и состоит из слоёв (layers). Каждая инструкция в Dockerfile создаёт новый слой.
- __Контейнер (container)__ — это запущенный экземпляр образа. Контейнер добавляет к образу записываемый слой, где сохраняются изменения во время работы.

### Docker Registry и Docker Hub

Где хранятся образы? В реестрах (registry). __Docker Hub__ — это публичный реестр, где сообщество и компании публикуют свои образы. Здесь вы найдёте официальные образы PostgreSQL, Nginx, Python и тысячи других.

Компании часто разворачивают приватные реестры для внутренних образов. Популярные решения: Harbor, GitLab Container Registry, AWS ECR.

Когда вы выполняете `docker pull postgres:16`, Docker сначала ищет образ локально. Не найдя его, обращается к Docker Hub и скачивает.

### Типичный сценарий работы

Допустим, вам нужна база данных PostgreSQL для анализа логов. Без Docker пришлось бы: скачать дистрибутив, установить, настроить пользователя, создать базу. С Docker весь процесс занимает одну команду.

```bash
# Скачать образ из Docker Hub
docker pull postgres:16

# Запустить контейнер
docker run -d --name my-postgres -e POSTGRES_PASSWORD=secret postgres:16

# Посмотреть запущенные контейнеры
docker ps

# Посмотреть логи
docker logs my-postgres

# Подключиться к контейнеру
docker exec -it my-postgres bash

# Остановить и удалить
docker stop my-postgres
docker rm my-postgres
```

Обратите внимание: после `docker rm` контейнер исчезает вместе со всеми данными внутри. Если данные нужно сохранить — используйте тома.

### Тома (volumes)

Данные внутри контейнера исчезают при его удалении. Чтобы сохранить данные, используйте тома:

```bash
# Именованный том (Docker управляет расположением)
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16

# Bind mount (вы указываете путь на хосте)
docker run -d -v /path/on/host:/var/lib/postgresql/data postgres:16
```

__Рекомендация:__ для баз данных используйте именованные тома. Для конфигурационных файлов — bind mount.

## Dockerfile

Что если нужного образа нет на Docker Hub? Или вы хотите добавить в образ свой код? Тогда вы создаёте собственный образ с помощью Dockerfile.

Dockerfile — это текстовый файл с инструкциями для сборки образа. Каждая инструкция создаёт новый слой. Docker кэширует слои, поэтому повторная сборка происходит быстро — пересобираются только изменившиеся слои.

### Пример: Python-приложение для анализа логов

```dockerfile
# Базовый образ
FROM python:3.11-slim

# Метаданные
LABEL maintainer="security-team"

# Рабочая директория
WORKDIR /app

# Копируем зависимости отдельно (для кэширования)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Копируем код приложения
COPY src/ ./src/

# Команда по умолчанию
CMD ["python", "src/analyzer.py"]
```

### Основные инструкции

- `FROM` — базовый образ
- `WORKDIR` — рабочая директория внутри контейнера
- `COPY` / `ADD` — копирование файлов в образ
- `RUN` — выполнение команды при сборке
- `ENV` — переменные окружения
- `EXPOSE` — документирование портов (не открывает их!)
- `CMD` / `ENTRYPOINT` — команда при запуске контейнера

### Best practices

- __Минимизируйте количество слоёв.__ Объединяйте команды RUN:

```dockerfile
# Плохо: 3 слоя
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# Хорошо: 1 слой
RUN apt-get update && \
    apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*
```

- __Используйте slim/alpine образы.__ `python:3.11-slim` весит ~150 МБ, `python:3.11` — ~900 МБ.
- __Не храните секреты в образе.__ Пароли и ключи передавайте через переменные окружения или Docker secrets.

### Распространённые ошибки

- __Копирование всего проекта перед установкой зависимостей.__ Любое изменение в коде инвалидирует кэш, и зависимости устанавливаются заново. Сначала копируйте только файл зависимостей.
- __Запуск от root без необходимости.__ Создавайте непривилегированного пользователя:

```dockerfile
RUN useradd -m appuser
USER appuser
```

## Docker Compose

Один контейнер — это хорошо. Но реальные системы состоят из нескольких компонентов: база данных, веб-сервер, очередь сообщений, система визуализации. Запускать каждый контейнер отдельной командой неудобно. Docker Compose решает эту проблему — вы описываете всю инфраструктуру в одном YAML-файле.

### Пример: PostgreSQL + Grafana

Допустим, вы строите систему мониторинга: PostgreSQL хранит события безопасности, Grafana визуализирует их. Вот как это выглядит в docker-compose.yml:

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: analyst
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: security_logs
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  grafana:
    image: grafana/grafana:latest
    environment:
      GF_SECURITY_ADMIN_PASSWORD: ${GRAFANA_PASSWORD}
    ports:
      - "3000:3000"
    volumes:
      - grafana-data:/var/lib/grafana
    depends_on:
      - postgres

volumes:
  pgdata:
  grafana-data:
```

Одна команда `docker compose up -d` — и вся инфраструктура работает.

### Ключевые элементы

- __services__ — определение контейнеров. Каждый сервис — это один контейнер.
- __environment__ — переменные окружения. Используйте `${VAR}` для подстановки из `.env` файла.
- __volumes__ — монтирование томов. Именованные тома объявляются в секции `volumes` внизу.
- __ports__ — проброс портов `"хост:контейнер"`.
- __depends_on__ — порядок запуска. Grafana запустится после postgres. Важно: это не гарантирует, что postgres готов принимать соединения, только что контейнер запущен.
- __networks__ — по умолчанию Compose создаёт общую сеть для всех сервисов. Сервисы обращаются друг к другу по имени (postgres, grafana).

### Команды Docker Compose

```bash
# Запустить все сервисы
docker compose up -d

# Посмотреть статус
docker compose ps

# Посмотреть логи
docker compose logs -f grafana

# Остановить и удалить
docker compose down

# Удалить вместе с томами
docker compose down -v
```

### Файл .env

Храните секреты в `.env` файле (добавьте его в `.gitignore`):

```
DB_PASSWORD=super_secret_password
GRAFANA_PASSWORD=admin_secret
```

## Альтернативы Docker

- __Podman__ — совместим с Docker, но не требует демона и может работать без root. Команды идентичны: `podman run`, `podman build`. Хороший выбор для систем с повышенными требованиями к безопасности.
- __containerd + nerdctl__ — containerd это runtime, который использует Docker под капотом. nerdctl — CLI-инструмент, совместимый с Docker. Используется в Kubernetes.
- __Kubernetes__ — система оркестрации контейнеров. Нужен, когда у вас десятки и сотни контейнеров, требуется автомасштабирование и высокая доступность. Для учебного курса избыточен.
- __LXC/LXD__ — контейнеризация на уровне ОС, ближе к виртуальным машинам. Контейнер ведёт себя как полноценная система с init. Используется, когда нужна полная ОС в контейнере.

## Практические задания

В директории `app/` находится Python-приложение `log_generator.py` — генератор событий аутентификации. Приложение имитирует реальную активность: обычные входы пользователей, периодические brute-force атаки, подозрительные подключения с внешних IP.

Ваша задача — упаковать это приложение в Docker-образ и развернуть инфраструктуру для его работы.

### Задание 1.1: Dockerfile для генератора

Напишите `Dockerfile` для упаковки приложения `log_generator.py`.

__Требования:__

- Используйте `debian:bookworm-slim` как базовый образ
- Установите `uv` для управления зависимостями Python
- Скопируйте `pyproject.toml` и `uv.lock` из корня репозитория
- Установите зависимости через `uv sync`
- Скопируйте код приложения
- Настройте запуск `log_generator.py` при старте контейнера

__Подсказки:__

- uv можно установить из официального образа: `COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv`
- Для запуска Python-скрипта через uv: `uv run python ...`

__Чеклист выполнения:__

- [ ] Dockerfile написан и находится в директории `lesson01/`
- [ ] Образ успешно собирается: `docker build -t log-generator .`
- [ ] Контейнер запускается без ошибок (пока без БД — будет ошибка подключения, это нормально)

### Задание 1.2: Docker Compose инфраструктура

Создайте `docker-compose.yml`, который разворачивает полную инфраструктуру.

__Сервисы:__

1. __postgres__ — база данных для хранения событий
2. __log-generator__ — ваш образ из задания 1.1
3. __grafana__ — визуализация данных

__Требования:__

- PostgreSQL:
    - База данных `security_logs`
    - Пользователь и пароль из переменных окружения
    - Именованный том для данных
    - Healthcheck для проверки готовности

- log-generator:
    - Собирается из локального Dockerfile
    - Запускается только после готовности postgres (используйте `depends_on` с `condition: service_healthy`)
    - Параметры подключения к БД через переменные окружения

- Grafana:
    - Пароль администратора из переменных окружения
    - Именованный том для данных

- Общее:
    - Все секреты в файле `.env`
    - Файл `.env` добавлен в `.gitignore`

__Переменные окружения для генератора:__

- `DB_HOST` — хост базы данных (имя сервиса в compose)
- `DB_PORT` — порт (5432)
- `DB_NAME` — имя базы данных
- `DB_USER` — пользователь
- `DB_PASSWORD` — пароль
- `EVENTS_PER_BATCH` — количество событий за раз (по умолчанию 10)
- `BATCH_INTERVAL` — интервал между батчами в секундах (по умолчанию 5)

__Чеклист выполнения:__

- [ ] `docker-compose.yml` описывает все три сервиса
- [ ] `.env` файл создан с необходимыми переменными
- [ ] `docker compose up -d` запускает инфраструктуру без ошибок
- [ ] Генератор подключается к базе и создаёт события
- [ ] В логах генератора видно: "Сгенерировано N событий"
- [ ] Grafana доступна на http://localhost:3000
- [ ] В PostgreSQL появляются данные в таблице `auth_events`

### Проверка результата

После запуска инфраструктуры подключитесь к PostgreSQL и проверьте данные:

```bash
docker exec -it <postgres_container> psql -U analyst -d security_logs -c "SELECT COUNT(*) FROM auth_events;"
```

Через несколько минут работы генератора в таблице должно быть несколько сотен записей.
